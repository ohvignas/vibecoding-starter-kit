# Synthèse — audit & axes de progression

## 1. Verdict global (5 lignes max)
Le cap est bon et validé par le marché : personne (ni BMAD, ni Agent OS, ni les kits anglo, ni l'offre FR) ne fait le flux maquette-validée → roadmap dérivée → build comparé à la maquette, en français, zéro dépendance, pensé pour une salle. Les forces uniques : maquette-first comme contrat de build, 3 stacks (Convex/Expo/Electron), jalons observables, posture sécurité outillée. Mais le risque n°1 est brutal : **le jour 1 en salle, le kit ne tient pas sa promesse** — projet généré sans `git init` (toute la sécu pre-commit est inerte), wizard qui gèle sur Ctrl+C, 100 % des élèves Windows bloqués (npx ENOENT, MinTTY), rapport qui affiche des faux ✅. La crédibilité du formateur se joue dans les 15 premières minutes ; tout le reste (pédagogie, Cursor 2026, features) vient après ce chantier de fiabilisation, majoritairement composé de correctifs S.

## 2. Ce qui est déjà fort (à valoriser en formation)
- **Le flux maquette-first Stitch → design.md → roadmap → build comparé à la maquette** : différenciateur absolu, à « breveter culturellement » avant que BMAD (DESIGN.md/EXPERIENCE.md) n'y arrive.
- **Le pattern « ✅ Ce que tu vois » par jalon** : chaque étape a un résultat observable — exactement ce que la recherche sur la motivation recommande.
- **`stacks/saas/prompts-de-demarrage.md`** : pédagogiquement remarquable (progressif, points d'arrêt, prompt de débogage) — aujourd'hui orphelin, à repositionner en leçon découverte officielle.
- **Zéro dépendance npm + hooks sécurité (guard-shell, scan secrets, propose-only)** : installable sans risque, argument fort face aux 45 % de code IA vulnérable et aux 170+ apps Lovable exposées (CVE-2025-48757).
- **Le socle Cursor est conforme aux docs de juillet 2026** : format .mdc, hooks.json v1, environment.json, BUGBOT.md, .cursorignore — RAS technique.
- **/doctor existe** : un vrai atout vs la concurrence, à muscler plutôt qu'à créer.
- **Le ton français (tutoiement, analogies) est excellent et cohérent** ; le glossaire est riche.
- **Le modèle de friction « 2 gestes manuels »** est la bonne North Star produit — il faut juste le rendre vrai (et le réduire à 2 clics).

## 3. Les axes d'amélioration, priorisés

### Axe 1 — Fiabiliser le wizard : qu'il fasse ce qu'il dit, et dise ce qu'il fait [priorité: P0] [effort: S]
- **Constat** : le wizard ne fait ni `git init` ni `git config core.hooksPath` (hooks et scan de secrets morts alors que le README les promet), écrase AGENTS.md/CLAUDE.md sans avertissement, affiche ✅ pour des fichiers non écrits, exit 0 malgré 22 échecs ENOENT lancé hors racine, gèle sur Ctrl+C (confirmé en pty), accepte `~/mon-app` (dossier littéral `~`), vérifie Node après les 5 questions (ou jamais sous Node 16), crée le projet dans le clone du kit, et n'a pas de mode non-interactif pour une cohorte de 20 élèves.
- **Reco** : une passe unique de correctifs S dans setup.mjs : git init + hooksPath + commit initial ; ne jamais écraser (AGENTS.md.new + signalement) ; rapport à 3 états (créé / conservé — vérifie-le / échec) et exit 1 si échecs ; handler SIGINT ; check Node en 1re ligne + readline en import dynamique ; source dérivée de `import.meta.url` ; défaut `../<nom>` avec chemin absolu affiché ; flags `--stack --assistant --project --yes` documentés + prompt final affiché dans tous les modes.
- **Findings sources** : « projet généré pas un dépôt git », « AGENTS.md écrasés sans avertissement » (×2), « rapport affiche ✅ même quand non écrit », « Ctrl+C ne quitte jamais », « hors racine : 22 ENOENT exit 0 », « core.hooksPath jamais configuré », « ~/mon-app accepté », « Node trop vieux », « nom relatif → app dans le clone », « mode flags/non-TTY », « pas de mode non-interactif », « sorties git brutes », « étape manuelle hooksPath automatisable », « .env vs .env.local ».

### Axe 2 — Windows : aujourd'hui 100 % cassé pour ces élèves [priorité: P0] [effort: M]
- **Constat** : toutes les installs de skills échouent (`execFileSync('npx')` = ENOENT depuis Node 20.12.2, le minimum exigé par le kit) ; sous Git Bash/MinTTY — le terminal par défaut que le guide 02 fait installer — le wizard ne se lance jamais et affiche un mur d'erreurs de flags ; caveman via `bash -lc` échoue aussi. Aucun runner windows en CI, donc invisible.
- **Reco** : `npx.cmd` + `shell: true` sur win32 dans defaultRun ; message dédié non-TTY (« lance depuis PowerShell ») ; PowerShell documenté comme terminal recommandé dans guide 02 ; matrice CI {ubuntu, windows, macos} × {20.12, 22} avec smoke E2E du setup (voir Axe 8).
- **Findings sources** : « Windows : installs skills échouent (ENOENT) », « Git Bash MinTTY : wizard jamais lancé », « CI du kit : pas de Windows, pas de Node 20.12 ».

### Axe 3 — Dépolluer Cursor et parler la langue de CHAQUE assistant [priorité: P0] [effort: M]
- **Constat** : trois enquêteurs indépendants confirment que le matching par tags déverse 64 à 201 règles .mdc hors-sujet (Angular, Solidity, Next.js, Vue…) avec `globs: **/*` dans un projet Convex+TanStack — pollution de contexte contradictoire avec la stack enseignée, l'anti-pattern exact des docs Cursor. En plus, les docs générées pour Cursor contiennent des commandes Claude Code (`/mcp`, `claude mcp add`, `.mcp.json`) : le débutant Cursor tape /mcp, rien ne se passe, bloqué à l'étape 3. Et `/add-plugin superpowers`, l'unique geste chat, n'est ni défini au glossaire, ni vérifié, ni doté d'un plan B.
- **Reco** : remplacer matchTags par une liste blanche de 3-10 fichiers vérifiés par stack (ou supprimer le clone : les règles typées du kit suffisent) ; rendre setup-ai.mjs et domains.mjs conditionnels par assistant ; ajouter plugin/superpowers au glossaire + ligne de vérification + fallback install manuelle ; passer le plugin Convex par `/add-plugin <slug>` marketplace au lieu d'un git clone.
- **Findings sources** : « 64 règles .mdc hors-sujet », « awesome-cursorrules 60+ règles » (robustesse), « ~201 règles .mdc » (cursor-etat-art), « commandes Claude Code dans docs Cursor », « /add-plugin superpowers sans définition », « plugin Convex : marketplace plutôt que clone ».

### Axe 4 — Time-to-first-win et chemin critique Stitch [priorité: P0] [effort: M]
- **Constat** : rien ne tourne à l'écran avant 6 phases gated (1-3 h après le clone) alors que la promesse est « tu vois ton produit grandir » ; la clé Stitch — le geste le plus technique du parcours (compte Google, clé API, JSON à coller) — est découverte trop tard, sans fallback : un élève sans compte ou à quota épuisé est coincé au milieu de /new-project. Le prompt final, artefact le plus critique du handoff, n'existe qu'en sortie terminal ; `--no-skills` affiche « ✅ skills déjà installés » à tort ; et /new-project référence des chemins du kit (`stacks/`, `templates/`) absents du projet généré, refait choisir une stack déjà figée et réécrit un AGENTS.md déjà posé.
- **Reco** : (1) fallback écrit dans Phase 5 : « pas de Stitch → l'IA génère des wireframes HTML dans maquette/ » (même contrat aval, Stitch devient enrichissement) ; (2) démo express hello-world scaffoldée+bootée en fin de wizard ou juste après le choix de stack (TTFW < 15 min) ; (3) prompt écrit dans COLLE-MOI-DANS-L-IA.md à la racine + report.mjs aligné sur la vraie séquence ; (4) new-project.md généré avec la stack injectée et « complète l'AGENTS.md existant » ; (5) deeplinks `cursor://` « Add to Cursor » pour les MCP (geste manuel → 1 clic, clé jamais commitée) ; (6) galerie maquette/index.html pour la validation d'un regard.
- **Findings sources** : « TTFW trop long : 6 phases gated », « clé Stitch sur le chemin critique sans fallback » (×2), « prompt final n'existe qu'en sortie terminal », « /new-project référence des chemins absents », « deux dossiers : confusion classique », « deeplink Add to Cursor » (×3), « time-to-first-win < 5 min : finir sur un wow », « galerie maquette/index.html ».

### Axe 5 — Anti-magie : un vrai parcours, un mode apprentissage, des preuves de compréhension [priorité: P0] [effort: M]
- **Constat** : c'est le cœur pédagogique manquant. Pas de progression leçon 1 → N (les guides s'arrêtent à 3, deux chemins coexistent sans articulation) ; ~30 artefacts générés sans explication du pourquoi (« magie subie ») ; aucun mécanisme n'oblige l'IA à expliquer ce qu'elle construit (`/build --all` encourage même le tout-automatique) ; zéro exercice ni checkpoint. Or la donnée marché est écrasante : les élèves qui questionnent le code généré scorent 65 %+ en compréhension vs <40 % pour les passifs — c'est LA variable prédictive du succès. S'ajoutent : ~15 docs sans hiérarchie « pour toi / pour l'IA », charge cognitive jour 1 (~10 gestes à plat dont plusieurs différables), PRD calibré entreprise (9 sections, 45-90 min de gates) pour un premier projet.
- **Reco** : PARCOURS.md séquencé (objectif, durée, livrable, critère de réussite par leçon) avec prompts-de-demarrage comme leçon découverte ; docs/VISITE-GUIDEE.md générée (3 lignes par artefact : c'est quoi / pourquoi / quand tu le verras agir) ; question wizard « apprendre en construisant ? » → règle AGENTS.md permanente (3 puces d'explication + 1 question de compréhension par jalon, /build --all désactivé) ; bandeaux « À lire par toi / Pour l'IA » + ONBOARDING en 2 blocs Maintenant/Plus tard avec déclencheurs ; option « projet d'apprentissage » = PRD 1 page ; checkpoints d'auto-évaluation en fin de guide ; projet d'échauffement 2 h avec 2 bugs volontaires avant le projet perso.
- **Findings sources** : « aucun parcours leçon 1 → N », « magie subie : ~30 artefacts », « pas de mode apprentissage transverse », « compréhension = variable prédictive (65 % vs 40 %) », « 7 docs sans hiérarchie », « surcharge conceptuelle jour 1 » (×2), « PRD calibré entreprise », « aucun exercice ni checkpoint », « prompts-de-demarrage orphelin », « scaffolding dégressif validé par la recherche », « projet d'échauffement guidé », « mini-spec 5 champs BMAD », « mode Rapide vs Coaché », « ONBOARDING trop court vs easy-vibe », « célébration des victoires », « frictions FR + CRUD faux au glossaire ».

### Axe 6 — Survivre à l'échec : doom loop, bouton panique, git comme assurance-vie [priorité: P1] [effort: M]
- **Constat** : le « doom loop » (IA corrige → nouveau bug → re-corrige) est le point d'abandon n°1 documenté, et 63 % abandonnent au « Month 3 Wall » ; or la gestion d'échec du kit est dispersée (le prompt de débogage n'existe que sur le chemin manuel), /doctor rend « tout est ✓ » à un élève dont l'app est cassée, et il n'existe ni retour arrière guidé (git revert reste terrifiant) ni modules debugging/déploiement que tous les cours de référence incluent.
- **Reco** : règle des 3 essais dans les règles Cursor (3 fixes ratés → stop, retour au dernier état vert, nouvelle conversation, BUG.md) ; /sos (résumé humain + 3 sorties : réparer / stash / revenir au dernier jalon vert) ; tags git `jalon-NN-nom` + /rewind en vocabulaire machine-à-remonter-le-temps ; guide 04-quand-ca-casse + docs/SOS.md générés ; /doctor avec étape 0 de triage ; git réduit à 3 gestes (checkpoint, retour arrière, historique) avec commit auto proposé par jalon vert ; ARCHITECTURE.md auto-maintenu + skill /debug et module déploiement par stack.
- **Findings sources** : « doom loop point d'abandon n°1 », « gestion de l'échec dispersée », « /doctor ne parle pas la langue d'un élève en panique », « git filet de sécurité, pas sujet avancé », « Month 3 Wall / ARCHITECTURE.md », « checkpoints nommés (Bolt) », « /sos bouton panique », « 2 modules manquants : debugging et déploiement », « docs vivantes auto-régénérées (VibeReady) ».

### Axe 7 — Sécurité réelle vs sécurité promise [priorité: P1] [effort: S]
- **Constat** : le scan de secrets pre-commit rate les formats que le kit lui-même fait manipuler (sk-ant-…, sk-proj-…, sk_live_ Stripe, re_ Resend — testés) ; guard-shell bloque `rm -f *.log` mais laisse passer `grep KEY .env`, et bloque l'installeur caveman du kit ; le guide sécurité enseigne le réflexe `curl | bash` sans avertissement ; la CI élève affiche une croix rouge au premier push (npm ci sans lockfile) ; checks.mjs risque d'installer le stub npm « tsc ». Contexte marché : 45 % du code IA échoue aux tests de sécurité de base.
- **Reco** : élargir la regex secrets (+ tests) avec option gitleaks ; resserrer guard-shell (ancres sur /, ~, * seul) + couvrir grep/awk/sed sur .env + beforeMCPExecution ; encadré pédagogique avant chaque `curl | bash` ; `npm ci || npm install` conditionnel ; exiger node_modules/.bin dans selectChecks ; skill /check-securite pré-déploiement par stack (règles d'accès Convex, secrets EAS, electron:security-audit).
- **Findings sources** : « scan de secrets rate les formats recommandés », « guard-shell : faux positifs », « curl | bash sans avertissement », « CI élève : npm ci sans lockfile », « checks.mjs npx tsc », « 45 % du code IA vulnérable — garde-fou outillé », « guard-shell vs sandbox natif 2026 ».

### Axe 8 — Vérification bout-en-bout et anti-pourrissement [priorité: P1] [effort: M]

## 4. Cursor 2026 — ce que le formateur doit savoir

**Conformité du kit : bonne.** Règles .mdc, hooks.json v1, environment.json, BUGBOT.md, .cursorignore sont conformes aux docs de juillet 2026 ; Cursor lit désormais nativement AGENTS.md (surveiller le doublon avec 00-project.mdc). Une erreur factuelle à corriger : Cursor interpole bien `${env:...}` dans les headers MCP (https://cursor.com/docs/context/mcp) — le choix user-scope pour Stitch reste défendable, mais pour la simplicité, pas par limitation.

**À adopter dans le kit** :
- **Skills (Cursor 2.4+)** : mécanisme central, `/migrate-to-skills` convertit commands et règles dynamiques ; Cursor lit `.claude/skills/` en rétrocompat — un seul format possible pour les deux assistants (https://cursor.com/docs/skills).
- **Deeplinks d'installation MCP** `cursor://...install?config=<base64>` : le geste manuel MCP devient 1 clic (https://cursor.com/docs/context/mcp/install-links).
- **Navigateur intégré de l'agent** (navigation, screenshots, console, sans MCP) pour la vérification visuelle de /build, et **Design Mode** (3.7) pour /edit-design (https://cursor.com/docs/agent/browser, https://cursor.com/changelog).
- **Plugins marketplace** : templates/cursor/ est déjà structuré comme un plugin ; `/add-plugin vibecoding-saas` remplacerait clone+wizard côté Cursor ; Team Marketplaces pour une cohorte (https://cursor.com/docs/reference/plugins).
- **Automations** (3.8, `/automate`) : équivalent natif du dream hook, notamment pour la consolidation mémoire (https://cursor.com/changelog/06-18-26).

**À enseigner en séance** : Tab ; Agent vs **Plan Mode (Shift+Tab)** — le flux PRD→roadmap→build du kit est une pédagogie plan-first, le montrer dans l'UI native (https://cursor.com/docs/agent/modes) ; la page **Customize** (3.9) où l'élève voit tous les artefacts posés par le wizard ; navigateur intégré + « clique l'élément » (l'UX Lovable dans Cursor) ; Bugbot (« bugbot run » en commentaire PR, coût usage-based à annoncer, https://cursor.com/docs/bugbot) ; les 3 couches de sécurité (sandbox natif + allowlist best-effort + guard-shell du kit, https://cursor.com/blog/agent-sandboxing) ; cloud agents /in-cloud et app iOS en démo « effet whaou ». Mettre en place une relecture du changelog (https://cursor.com/changelog) avant chaque session.
